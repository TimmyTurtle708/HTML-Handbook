🧠 心理建設（先看這裡）
1
你學的 HTML 在這裡為什麼不能用？ ►

這是每個新手最大的困惑，也是最重要的心理建設。 你在外面學的現代 HTML/CSS，預設對象是「瀏覽器」。瀏覽器 Chrome、Firefox、Safari 都很努力跟上 Web 標準，支援 flexbox、grid、CSS 變數、動畫⋯⋯ 但 Email 的對象是「郵件客戶端」，而每個客戶端各自解讀 HTML 的方式都不一樣，而且差異大到嚇人。

Email HTML ≠ 網頁 HTML。這不是你學的東西有問題，而是 Email 這個媒介本身就活在另一個時代。
2
Outlook 為什麼這麼難搞？（歷史背景） ►

Outlook 2007 之後微軟做了一個影響深遠的決定：改用 Microsoft Word 當作郵件的渲染引擎。 這意味著什麼？Word 是文字處理軟體，不是瀏覽器。它的 CSS 支援停留在非常原始的程度，而且一直沒有大幅改善。 • Outlook 2007–2019：用 Word 渲染，大量 CSS 失效 • Outlook 365（新版）：已改用 Edge WebView，好很多 • 但很多企業用戶還在用舊版 Outlook 所以你沒有選擇，只能為最差的環境設計。

caniemail.com 可以查每個 CSS 屬性在各客戶端的支援程度，是 Email 工程師的必備工具。
3
「我明明沒改什麼，怎麼就壞了？」 ►

Email 開發最崩潰的地方：同樣的 code 在不同地方看起來完全不同。 • 你的 Gmail 網頁版看起來正常 • 同事的 Outlook 桌機版完全跑版 • 手機 Gmail app 又是另一個樣子 • 深色模式下顏色全部反轉 這不是你的錯，這是 Email 這個環境的本質。請把「在所有地方都完美」這個期望放下，改成「在主要平台都可以接受」就好。

永遠不要只看自己的瀏覽器就交件。至少要用實際手機確認一次。
4
用「降級思維」設計 ►

正確的心態是：先讓最爛的環境勉強能看，再讓好的環境看起來漂亮。 ✅ 用 table 做排版 → 所有地方都能跑 ✅ 用 inline CSS → 不怕被 email client 濾掉 ✅ 圖片加 alt 文字 → 圖片被擋時還有說明 ✅ 背景圖加底色 fallback → 不支援背景圖也有顏色 不是退步，是針對環境的專業選擇。

把每個設計決策想成「如果這個東西不支援，會不會整個壞掉？」，如果會，就要有備案。
5
新手最常有的誤解 ►

❌「應該有更優雅的寫法」 → 有時候真的沒有，古老的暴力解法就是最穩的解法。 ❌「這段 code 好醜，能不能重構一下」 → Email HTML 的 code 本來就很醜，醜是正常的。 ❌「這個問題 Google 搜尋找不到解答」 → 找 StackOverflow 搜「html email」相關，或去 litmus.com 社群找，資源比你想像的多。 ❌「測試環境沒問題就代表沒問題」 → 永遠不夠，要在實際收件環境測試。

📐 Email HTML 的世界規則
1
規則 1：排版只能用 table ►

Email 不能用 div + flexbox，必須用 table 建構。一個 Block 就用一個 table 包，Block 間距用 td 的 padding（margin 在 table 上不成立）。

<table width="700" border="0" cellspacing="0" cellpadding="0"> <tr><td>內容</td></tr> </table>
table 是 Email 的 flexbox，tr 是列，td 是欄。一旦接受這件事，剩下的都是體力活。
